## Summary

On June 27, 2026, we handled another CRAN follow-up for `jpmap`. CRAN reported
possibly invalid README article links such as `articles/getting-started.html`,
noted a 16,276,870-byte tarball, and asked whether functionality and map data
could be split into separate packages so map data would not be repeatedly
stored on CRAN mirrors.

The canonical Dropbox repository already contained the intended functionality
and data split: large boundary GeoPackages had been removed from `jpmap`, and
boundary discovery used the companion `jpmapdata` package or locally built MLIT
N03 files. The remaining practical issue was that map-heavy vignettes could
embed large outputs when optional boundary data were installed on the build
machine, especially the Leaflet vignette. The vignette setup was tightened so
heavy map chunks run only for pkgdown builds or when
`JPMAP_FULL_VIGNETTES=true`. For CRAN source builds, those chunks see an empty
temporary data directory and do not embed the large map or htmlwidget outputs.

The README inside the rebuilt source tarball was checked directly and contains
absolute pkgdown article URLs. The rebuilt tarball is about 154 KB, and the
Leaflet vignette HTML is about 15 KB rather than a multi-megabyte embedded
widget. After reading all existing `project_history/*.md` files, the package
version was bumped to `0.1.2` for the fresh CRAN resubmission. The final
artifact is `jpmap_0.1.2.tar.gz`.
