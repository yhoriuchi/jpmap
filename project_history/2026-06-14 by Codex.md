**Codex response:** Updated the pkgdown source and generated website. Added
top-page logo spacing, GitHub install instructions, top-page vignette links,
and singular starter calls such as `plot_jpmap("prefecture")` and
`plot_jpmap("municipality", include = "Okinawa")`. Added official MLIT N03
2024 Okinawa municipal boundaries as bundled data and changed data selection
so default prefecture maps still use all-Japan boundaries while Okinawa
municipal maps automatically use the bundled Okinawa file. Replaced the old
combined/broad articles with clearer download/import, transform, inset, and
four simple plotting tutorials. Rebuilt the pkgdown site after cleaning stale
generated pages, verified the site in the in-app browser, and confirmed
`R CMD check --no-manual` on the built source tarball finished with
`Status: OK`.

Follow-up fix: changed the "Where Files Are Saved" chunk in the Download
Boundary Data vignette so the rendered website does not print the maintainer's
machine-specific `jpmap_data_dir()` path. Rebuilt pkgdown and verified no local
user path remains in the public docs/search output.

Map readability follow-up: changed `plot_jpmap()` defaults to use a gray
ggplot-style panel, darker outlines, a visible default fill, longitude/latitude
graticules, and projected rectangular inset boxes. Updated the prefectural and
municipal tutorial figures to use higher-contrast map styling. The Okinawa
municipal choropleth and point tutorials now focus on Okinawa Island and filter
out offshore polygon fragments so the base examples are easier to read.

Inset graticule follow-up: masked the main map graticules inside transported
inset boxes and added local true-coordinate longitude/latitude lines and labels
for Okinawa and Ogasawara. Updated FAQ and inset documentation to clarify that
the labels inside boxes are true island coordinates, not destination
coordinates.

Inset extent follow-up: enlarged the Okinawa and Ogasawara boxes from source
extents so transported islands fit inside the visual frames. Retuned the
Ogasawara inset scale so remote pieces such as Minamitorishima fit in a compact
box, and narrowed the default map x-limits around the resulting layout. The
plot limits now use only the inset regions actually shown, so Okinawa-only maps
do not reserve space for a hidden Ogasawara box.

Inset placement/API follow-up: moved Okinawa back to the upper-left inset
position and kept Ogasawara in the lower-right position. Reworked inset label
placement so latitude labels sit on their true graticule lines without
touching box borders. Added `plot_jpmap()` controls for longitude/latitude
limits, breaks, and labels, plus a vignette example for showing the main
islands with Okinawa and no Ogasawara inset.

Inset layout tightening follow-up: moved Okinawa farther toward the upper-left
corner and shifted Ogasawara upward and left from the lower-right frame.
Changed the default inset map frame to use tight projected limits from the
rendered map and inset boxes while keeping user-supplied limits and axis
breaks in longitude/latitude units. Lowered the Ogasawara box so its bottom
edge sits close to the bottom of the figure without touching it, better
matching the Okinawa box's tight top/left spacing.

Territorial-dispute follow-up: added disputed-territory shape support to
`jp_map()`, `plot_jpmap()`, and the new `jp_disputed_territories()` helper.
The package lets users exclude these areas with `territorial_disputes = FALSE`
or include only selected shapes such as `"senkaku"` and `"takeshima"`. The
Ogasawara inset expands when the Okinotorishima shape is included.

Disputed-territory geometry follow-up: replaced the initial circular markers
with packaged island/reef shapes. Natural Earth 1:10m land polygons provide the
Northern Territories and Senkaku shapes, and OpenStreetMap geometry via Overpass
provides Okinotorishima and Takeshima / Liancourt Rocks. The package removes
any disputed shapes from the bundled base layer before adding the curated
disputed-territory layer, so inclusion/exclusion stays consistent.

Disputed-territory styling follow-up: changed `plot_jpmap()` so disputed
territory shapes can use ordinary map styling. Added optional emphasis controls
such as `disputed_fill`, `disputed_color`, and `disputed_dots`.

Package-positioning and web-map follow-up: surveyed related Japan map packages
and documented how `jpmap` differs from `NipponMap`, `jpndistrict`,
UchidaMizuki's `jpmap`, `kokudosuuchi`, `jpcity`, and mesh/grid packages.
Added `jp_map_join()` for safer joins by Japanese administrative codes and
names, including numeric prefecture codes such as `1` matching `"01"`. Added
`jp_map_leaflet()` and a Leaflet vignette so `jpmap` can produce interactive
website maps while keeping `plot_jpmap()` focused on static inset maps.

Logo, Leaflet, and native-pipe follow-up: regenerated the package sticker from
the current rectangular-inset Japan map, added a live Leaflet example to the
website, and revised user-facing code examples to prefer tidyverse workflows
with the native R pipe `|>`.

Samurai Blue follow-up: changed the website theme, map examples, Leaflet
palette defaults, disputed-territory highlight examples, and regenerated logo
assets to use a Samurai Blue-inspired palette instead of the earlier
burgundy/gold palette.

CRAN-prep follow-up: prepared release metadata for a first CRAN submission by
setting the version to 0.1.0, adding `Authors@R`, excluding pkgdown and
submission-helper files from source builds, adding `inst/COPYRIGHTS` for
bundled boundary/sample data, and drafting `cran-comments.md`.

Badge and example follow-up: removed premature CRAN/download badges before the
package is on CRAN, kept the GitHub Actions check badge, removed public
wording that implied disputed-territory handling was hidden, switched the
municipal choropleth vignette to 2020 Okinawa Island municipality population,
and used `ggthemes::scale_color_colorblind()` for the municipal point map.

CRAN readiness and municipal-default follow-up: verified that all exported
functions and bundled datasets have Rd aliases, added source roxygen blocks for
the remaining exported functions, and confirmed Rd checks are clean. Changed
`plot_jpmap("municipality", include = "Okinawa")` so filtered municipality
plots default to a local non-inset frame unless `inset` is supplied explicitly.
Added `.github` to `.Rbuildignore` and reran `R CMD check --as-cran`; the final
local result was 0 errors, 0 warnings, and 2 notes: the expected new
submission/package-size note and a local HTML Tidy version note.

CRAN resubmission split follow-up: after CRAN asked that large map data be
separated from the functionality package, removed the large boundary
GeoPackages from `jpmap`, made boundary discovery use the companion
`jpmapdata` package or user-built local files, guarded examples and vignettes
so `jpmap` checks cleanly without boundary data installed, and updated README,
vignettes, help pages, copyrights, and `cran-comments.md` to describe the split.
Started a sibling `jpmapdata` package to hold boundary GeoPackages and source
notices, including the existing 2021 prefecture example and 2024 Okinawa
municipal files. Built a national 2024 MLIT all-municipality GeoPackage as a
size test; it succeeded but was about 354 MB uncompressed and roughly 223 MB
gzipped, so it was kept out of the first CRAN-facing `jpmapdata` package.

Version-bump follow-up: bumped `jpmap` from 0.1.0 to 0.1.1 before CRAN
resubmission so the corrected source tarball has a new package version.

CRAN 0.1.1 resubmission follow-up: the resubmitted `jpmap_0.1.1.tar.gz`
passed CRAN incoming pretests with only the normal new-submission note and
spelling flags for real technical/geographic terms (`GeoPackage`, `MLIT`, and
`Ogasawara`). The earlier invalid README article URI note and the large-tarball
note were gone. The package is pending manual CRAN inspection; no further
resubmission should be made unless CRAN requests another change.

Companion data-package follow-up: created and pushed the sibling
`jpmapdata` repository at `https://github.com/yhoriuchi/jpmapdata.git`. The
package contains the boundary GeoPackages that were removed from `jpmap`, plus
metadata and repository links. The all-Japan 2024 municipal GeoPackage was
tested but left out because it is far too large for a first CRAN data package.
The plan is to wait for CRAN feedback on `jpmap` before submitting
`jpmapdata`.

Website post-resubmission follow-up: rebuilt and committed the `jpmap` pkgdown
site after the data split so the public website explains `jpmapdata`, no longer
prints local installation paths, and reflects version 0.1.1. A small
website-only roadmap note was added to the Boundary Data Policy page explaining
that electoral districts, historical boundaries, and other special geographies
should generally be user-supplied rather than bundled in `jpmap` or
`jpmapdata`; future helpers can make these easier to read, check, transform,
and plot.

Changelog follow-up: restored the `jpmap 0.1.0` changelog entry for the full
initial package feature list and made `jpmap 0.1.1` a short CRAN
resubmission/data-split entry. Regenerated the website changelog and search
index so the public site shows both versions correctly.
